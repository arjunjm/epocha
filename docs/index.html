Features

Everything you need to
learn history deeply

Epocha combines AI generation with interactive tools to make history stick.

⚡

Instant AI Timelines

Type any topic and Claude generates a comprehensive, event-by-event timeline — complete with context, significance, tags, and key figures. Pre-built topics load in under 50ms; new topics stream live from Claude in 15–90 seconds.

🧠

Knowledge Quizzes

Every timeline comes with a 5-question MCQ quiz generated by Claude, with explanations. Earn XP, track progress, and unlock themes from the Marketplace.

🗺️

Learning Paths

5 curated journeys through connected topics — Ancient World, Rise of Science, Age of Empires, Age of Revolutions, Exploration & Discovery — with per-step progress tracking.

📌

Bookmarks & Library

Save timelines and bookmark individual events. Your library persists across sessions. Bookmarks grouped by topic with Markdown export and reading-progress tracking.

🔥

Trending Topics

Trending timelines are pre-generated by an Azure Function and served from Redis instantly. No waiting — click and read in milliseconds.

🃏

Flashcard Study Mode

Any timeline becomes spaced-repetition flashcards. Score yourself ✓/✗ per card, retry missed events, and exit with a full score breakdown.

📊

Timeline Insights

Events-by-century bar chart, top figures, themes, and key locations — all computed client-side from the cached JSON with proportional bars.

🗺 Density

Event Density Heatmap

Colour-coded grid bucketing events by era. Bucket size auto-scales: year/decade/century/millennium based on the timeline's span. Hover to inspect, click to scroll.

AI pipeline

How Claude builds a timeline

Every generation passes through a structured pipeline — from prompt construction to JSON validation and Redis storage.

01
Prompt Construction
System prompt injected with topic, time period, and output schema. cache_control: ephemeral applied — Anthropic caches the system prompt for 5 minutes, saving ~80% of input tokens on repeated calls.
prompt cache
02
Adaptive Thinking
claude-opus-4-7 uses extended thinking tokens to plan the event selection and chronological structure before generating output, improving factual accuracy and coverage.
thinking tokens
03
Streaming JSON
Output arrives as a single JSON object streamed token-by-token. The server accumulates chunks via SSE, forwarding raw text to the client while assembling the full payload.
SSE stream
04
Parse & Validate
On stream close, the server JSON.parse()s the accumulated text. Schema validation checks for required fields (topic, period, events array). Corrupt entries are rejected.
schema guard
05
Cache & Serve
Valid JSON is stored in Redis with a 7-day TTL and the normalised cache key. The "complete" SSE event fires with the full timeline — all future requests for this topic return in <50ms.
Redis setex 604800
Cache strategy

Redis key design

All timeline and quiz data lives in Azure Cache for Redis. Key normalisation ensures consistent hits across different capitalisation and whitespace.

Timeline cache key
timeline:{topic}:{startYear}:{endYear}
Topic is lowercased and spaces replaced with hyphens.
Years are stored verbatim (e.g. 27 BCE, 476 CE, 1947).
TTL: 7 days · eviction: allkeys-lru
timeline:the-roman-empire:27 BCE:476 CE
Quiz cache key
quiz:timeline:{topic}:{startYear}:{endYear}
Same normalisation as timeline keys, with quiz: prefix.
Validation guard rejects corrupt entries (e.g. bare string arrays instead of MCQ objects) and deletes them so Claude regenerates cleanly.
5 random questions served per session from a pool of 12.
quiz:timeline:the-roman-empire:27 BCE:476 CE
Trending topics sorted set
epocha:trending-topics
Redis sorted set — no TTL. Each member is a JSON string {topic, startYear, endYear, period}, score = view count.
Pre-generated by the Azure Function every 6 hours.
Server extracts years from the period field for old entries without explicit year fields.
Empty-year fallback
timeline:{topic}::
Legacy entries generated before year-keying was added live at the empty-year key (topic::). On a miss for the specific-year key, the server checks the empty-year key and lazily migrates the data to the correct key — so old cache entries still serve without regeneration.
